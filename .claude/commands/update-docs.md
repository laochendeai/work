## 文档分析

1. 审查当前文档状态：
   - 检查 `specs/implementation_status.md` 获取整体项目状态
   - 审查已实施阶段文档（`specs/phase{N}_implementation_plan.md`）
   - 审查 `specs/flutter_structurizr_implementation_spec.md` 和 `specs/flutter_structurizr_implementation_spec_updated.md`
   - 审查 `specs/testing_plan.md` 确保其针对最近的测试通过、失败和更改保持最新
   - 检查 `CLAUDE.md` 和 `README.md` 的项目级文档
   - 检查并记录 CLAUDE.md 中任何新的经验教训或最佳实践

2. 分析实施和测试结果：
   - 审查上一阶段实施的内容
   - 审查测试结果和覆盖率
   - 识别实施过程中发现的新最佳实践
   - 注意任何实施挑战和解决方案
   - 交叉引用更新的文档与最近的实施和测试结果以确保准确性

## 文档更新

1. 更新阶段实施文档：
   - 用 ✅ 状态标记已完成的任务
   - 更新实施百分比
   - 添加实施方法的详细说明
   - 记录任何与原计划的偏差并说明理由
   - 如需要添加新章节（经验教训、最佳实践）
   - 为复杂组件记录具体实施细节
   - 包含阶段期间发现的任何新故障排除提示或工作流改进的摘要

2. 更新实施状态文档：
   - 更新阶段完成百分比
   - 添加或更新组件的实施状态
   - 添加实施方法和决策的说明
   - 记录实施期间发现的最佳实践
   - 注意任何克服的挑战和实施的解决方案

3. 更新实施规范文档：
   - 用 ✅ 或删除线标记已完成的项目但保留原始要求
   - 在适当位置添加实施细节的说明
   - 添加对已实施文件和类的引用
   - 根据经验更新任何实施指导

4. 如需要更新 CLAUDE.md 和 README.md：
   - 添加新的最佳实践
   - 更新项目状态
   - 添加新的实施指导
   - 记录已知问题或限制
   - 更新使用示例以包含新功能

5. 记录新的测试程序：
   - 添加创建的测试文件的详细信息
   - 包含测试运行说明
   - 记录测试覆盖率
   - 解释复杂组件的测试方法

## 文档格式和结构

1. 保持一致的文档风格：
   - 使用清晰的标题和章节
   - 在有帮助的地方包含代码示例
   - 一致使用状态指示器（✅, ⚠️, ❌）
   - 保持正确的 Markdown 格式

2. 确保文档完整性：
   - 涵盖所有已实施的功能
   - 包含使用示例
   - 记录 API 更改或添加
   - 包含常见问题的故障排除指导

## 指导原则

- 不要创建新的规范文件
- 更新 `specs/` 目录中的现有文件
- 保持一致的文档风格
- 在适当的地方包含实际示例
- 交叉引用相关文档章节
- 记录最佳实践和经验教训
- 提供项目进度的清晰状态更新
- 更新数字完成百分比
- 确保文档反映实际实施

完成后提供文档更新的摘要，包括：
1. 更新的文件
2. 文档的主要更改
3. 更新的完成百分比
4. 记录的新最佳实践
5. 此阶段后整体项目的状态
